# Internationalization (EN ↔ BN) — Architecture & Rollout Plan

Toolkit should support English (`en`) and Bengali (`bn`) across the whole Wasp
app (landing page, auth, account, admin, and all subapps: `carely`,
`reddit-bot`, `online-judge`, `sokafilm`).

This document is the design of record. Treat it as a living spec — update it
when the decisions below change.

## 1. Library choice

We use the standard React i18n stack — no hand-rolled context provider.

- [`i18next`](https://www.i18next.com/) — core engine (interpolation, plurals,
  namespaces).
- [`react-i18next`](https://react.i18next.com/) — `useTranslation` hook, `<Trans>`.
- [`i18next-browser-languagedetector`](https://github.com/i18next/i18next-browser-languageDetector)
  — detect + persist the chosen language in the browser.
- [`i18next-http-backend`](https://github.com/i18next/i18next-http-backend)
  — lazy-load per-namespace JSON over HTTP.
- [`i18next-parser`](https://github.com/i18next/i18next-parser) **(dev only)**
  — extract `t('…')` calls to keep JSON files in sync with code.

Rejected alternatives:

- **`react-intl` / FormatJS** — more ceremony, ICU-first. Overkill for our
  mostly simple static copy.
- **`@lingui/js`** — requires babel/extractor integration in the Wasp/Vite
  pipeline. More moving parts for no extra win here.
- **Custom React Context + maps** — fine for 10 strings, unworkable at our
  size and subapp count. Would be literally reinventing the wheel.

## 2. File / folder layout

Translations are **colocated with the code that uses them** (per subapp), so
moving/deleting a subapp moves/deletes its translations. Only genuinely global
copy lives in `common`.

```
app/src/i18n/
  index.ts                 # i18next bootstrap (imported once in App.tsx)
  LanguageSwitcher.tsx     # EN/BN toggle component
  locales/
    common/                # nav, auth, generic buttons, error messages
      en.json
      bn.json
    landing/
      en.json
      bn.json

app/src/carely/i18n/
  en.json
  bn.json

app/src/reddit-bot/i18n/
  en.json
  bn.json

app/src/online-judge/i18n/
  en.json
  bn.json

app/src/sokafilm/i18n/
  en.json
  bn.json

app/src/admin/i18n/
  en.json
  bn.json
```

Namespaces used by `useTranslation(...)`:
`common`, `landing`, `carely`, `reddit-bot`, `online-judge`, `sokafilm`, `admin`.

### 2.1 Bundling strategy

Hybrid so both web and the Capacitor Android shell stay fast:

- **`common`** is imported **statically** (small, always needed) → lands in
  the initial JS bundle. No network round-trip for core UI. Stored as
  `.ts` modules (not JSON) because the Wasp SDK tsconfig disables
  `resolveJsonModule`; semantics are identical — the modules just
  `export default` the translation object.
- **Per-subapp namespaces** are authored as `.json` files (so
  `i18next-parser` can manage them and non-engineers can PR translations)
  and lazy-loaded over HTTP via `i18next-http-backend` from
  `/locales/{ns}/{lng}.json`.
- The Vite plugin at `app/src/i18n/vitePlugin.ts` discovers every
  `src/**/i18n/*.json` at build time and emits each as an unhashed asset
  so they land at `/locales/<ns>/<lng>.json` in the Nginx-served bundle.
  During `wasp start`, the same plugin serves them via a dev-server
  middleware so edits hot-reload without a build step.

## 3. Initialization

`app/src/i18n/index.ts`:

- `i18n.use(HttpBackend).use(LanguageDetector).use(initReactI18next).init(...)`
- `fallbackLng: 'en'`, `supportedLngs: ['en', 'bn']`, `defaultNS: 'common'`.
- Detector order: `['cookie', 'localStorage', 'navigator']`.
- Persistence caches: `['cookie', 'localStorage']`.
- Imported **once** at the top of `app/src/client/App.tsx`.

## 4. Language persistence

### 4.1 Default: client-only (Phase 1)

- Detector/storage keys: `i18nextLng` in both `localStorage` and a cookie.
- On toggle: `i18n.changeLanguage(next)` → detector writes to both stores.
- On next cold start: detector reads them back; no server round-trip needed.
- Also set `document.documentElement.lang = next` for a11y / spellcheck.

**No server changes. No schema migration.**

### 4.2 Optional later: server echo (Phase 2+)

When we want emails (nodemailer templates, password reset, invites) in the
user's language:

- Add `locale String @default("en")` to the `User` Prisma entity.
- Add Wasp action `setUserLocale(lang)` — fire-and-forget on toggle.
- On login, sync with the value the server returned (if present).

Kept explicitly out of scope for Phase 1 to avoid a migration.

## 5. Does adding a new translation require an app rollout?

Two deploy artifacts to keep distinct:

| Artifact | Rebuild needed for new/changed translation? |
| --- | --- |
| **Wasp backend** (`toolkit`) | No |
| **Wasp frontend** (`toolkit-ui`) | Yes (fast, isolated; this is the *only* place strings live) |
| **Mobile app APK** (Capacitor Android) | **No** (as long as we stay in hosted mode) |

Rebuilding the **Wasp frontend** is acceptable. Avoiding **APK rebuilds** is
the explicit goal — and our current architecture already gives us that.

### 5.1 Why no APK rollout is needed (today)

`mobileapp/capacitor.config.ts` is in **hosted mode**:

- `server.url = https://toolkit.naurinjahan.com/carely` (see
  `mobileapp/capacitor.config.ts`).
- The APK ships almost no content of its own; the Android WebView loads the
  deployed frontend at runtime.
- Therefore: when we redeploy the **frontend** (`toolkit-ui`), the updated
  `/locales/**` JSON is picked up by the mobile app on the next WebView
  fetch — **no APK rebuild, no Play Store release, no user update**.

This property is why translations should stay on the frontend origin:
adding a translation = push frontend → mobile picks it up.

### 5.2 What would break this promise (avoid)

- **Switching Capacitor to bundled/offline mode** (i.e. removing
  `server.url` and shipping the built web app inside the APK). Then
  translation JSON lives in the APK and each change needs an APK rollout.
  If we ever do offline support, we must keep translations loaded from the
  remote origin (or from a remote `/locales/**` CDN; see 5.3) rather than
  bundling them.
- **Loading locales from a bundled import** (e.g.
  `import en from './locales/en.json'` baked into the JS bundle). That
  works for web but re-baking the bundle defeats the purpose. Keep lazy
  HTTP loading via `i18next-http-backend`.

### 5.3 Cache behavior (relevant for mobile)

The Android WebView can be sticky about caching JSON responses. This is
already handled **centrally** by the nginx config generated in
`app/deploy.sh` — no i18n-specific work is needed on the server side:

| Path            | `Cache-Control`                                | Why                                             |
| --------------- | ---------------------------------------------- | ----------------------------------------------- |
| `/assets/*`     | `public, max-age=31536000, immutable`          | Vite content-hashes these; never collide.       |
| `/locales/*`    | `public, max-age=300, must-revalidate`         | Translations update without a frontend rebuild. |
| `/index.html`   | `no-store, no-cache, must-revalidate`          | Entry point must always be fresh.               |
| `/sw.js` et al. | `no-store, no-cache, must-revalidate`          | Service workers can pin users to stale UI.      |
| everything else | `public, max-age=300, must-revalidate`         | Favicon/og-image-style public assets.           |

Combined with `i18next-http-backend`'s built-in `queryStringParams: { v: BUILD_HASH }`
(recommended), a frontend rollout either:

1. Changes the `?v=` query string so WebViews consider it a new URL, **or**
2. At worst makes users wait ≤ 5 minutes (the `must-revalidate` TTL) before
   new translations appear.

Net effect: a user on the mobile app sees new strings within the cache
window (or immediately on the next cold start), with **no APK rollout**.

### 5.4 Going even further (optional, not Phase 1)

If we ever want translation edits without even a **frontend** rollout:

1. **External static hosting for `/locales/**`.** Point
   `i18next-http-backend.loadPath` at a CDN/S3 bucket we control. Editing
   a JSON there propagates within the CDN TTL — no Wasp rebuild.
2. **DB-backed translations + admin CRUD UI.** Store key/value rows in
   Postgres, expose a Wasp operation, feed them to i18next as an in-memory
   backend. Heavy; only justified if non-engineers edit copy regularly.

**Recommendation:** stick with "rebuild frontend" in Phase 1. The mobile
APK rollout is what we actually needed to avoid, and the hosted-mode
architecture already handles that.

## 6. Language switch UI

Matches the shape of the existing `DarkModeSwitcher`.

`app/src/i18n/LanguageSwitcher.tsx`:

- Two-language toggle (not a dropdown) — one click flips EN ↔ BN.
- Label: `EN` / `বাং` (Bengali abbreviation).
- Uses `useTranslation()` → `i18n.changeLanguage(next)`.
- Subscribed components auto-rerender through `react-i18next`.
- Sets `document.documentElement.lang`.

Placement in `app/src/client/components/NavBar/NavBar.tsx`:

- **Desktop top nav** (`NavBarDesktopUserDropdown`) — sibling of
  `DarkModeSwitcher` near line 128.
- **Mobile** (`NavBarMobileMenu`) — visible alongside the burger
  `SheetTrigger` (outside the sheet so it's reachable without opening the
  menu), and also repeated **inside** the sheet next to `DarkModeSwitcher`
  near line 233 so users who open the menu see it too.

## 7. Coding conventions (enforced in reviews)

These rules keep the i18n migration shippable incrementally:

- **Always use `t('key')`.** Never string-concatenate translated fragments.
- **Use `<Trans>`** for copy with embedded components (e.g. a link inside a
  paragraph). Do not template React nodes into `t()`.
- **Interpolate variables,** never concatenate:
  `t('greeting', { name })` ✅, not `` `${t('greeting')} ${name}` `` ❌.
- **Plurals** via i18next counts: `t('item', { count })`. CLDR plural rules
  already cover both `en` and `bn`.
- **Numbers / dates** go through a tiny helper using
  `Intl.NumberFormat` / `Intl.DateTimeFormat` seeded with `i18n.language`.
  Centralize this — no scattered `new Intl.…` calls.
- **Keys are hierarchical** (`carely.measurements.addButton`), not sentence
  text. Sentence-keys make refactoring painful.
- **Do not translate server-emitted errors inline.** Return an error *code*
  from Wasp ops (e.g. `CARELY_NOT_FOUND`) and map in the client:
  `t('errors.CARELY_NOT_FOUND')`. No server-side i18n runtime needed.

## 8. Cross-cutting concerns

- **Bengali webfont.** Add Google Fonts **Hind Siliguri** (or **Noto Sans
  Bengali**) via `@import` in `Main.css`. Without this, Bengali renders with
  the system fallback and looks poor on Windows and older Android WebViews.
- **Namespace suspense.** Lazy-loaded namespaces can briefly render raw
  keys on first visit. Fix with either:
  - `<Suspense>` boundary around subapp routes, or
  - preload namespaces on nav link hover, or
  - always include that subapp's namespace in the initial bundle if it is
    the landing route (e.g. `/carely` on the mobileapp).
- **Translation drift.** Run `i18next-parser` in CI / pre-commit to flag
  missing or stale keys across `en.json` ↔ `bn.json`.
- **SSR.** N/A — we don't SSR today. Skip `resources-to-backend` setups.

## 9. Rollout plan

Strict rule: **no big-bang translate-everything PR.** Ship in waves so each
lands independently and nothing regresses.

### Phase 1 — Infra only (no visible copy change)

- Install deps (`i18next`, `react-i18next`,
  `i18next-browser-languagedetector`, `i18next-http-backend`, dev-only
  `i18next-parser`).
- Add `src/i18n/index.ts`, import once in `App.tsx`.
- Add `LanguageSwitcher`; mount in desktop nav + mobile burger row + inside
  the mobile sheet.
- Seed `common/en.json` and `common/bn.json` with ~20 universal strings
  (Log in, Log out, Save, Cancel, Delete, Settings, Account, Loading, core
  error messages).
- Add `public/locales/` sync step to the build.
- Add Bengali webfont in `Main.css`.

Untranslated pages keep rendering their existing English — that is the
entire point of namespaces + `t()` with English fallback.

### Phase 2 — Translate `common` use-sites

Replace hardcoded strings in `NavBar`, `UserDropdown`, `AccountPage`,
`LoginPage`, `SignupPage` with `t(...)`.

### Phase 3 — Per subapp, one PR each

Order by user value:

1. `carely` (largest user-facing feature).
2. `landing` + `auth` (first-time user path).
3. `reddit-bot`, `online-judge`, `sokafilm`, `admin` (any order).

### Phase 4 (deferred) — Server-side locale

Only if/when we localize outgoing emails. Adds `User.locale`,
`setUserLocale` action, localized nodemailer templates.

## 10. Risks / gotchas

- **Key leakage on cold start.** Mitigated by including `common` in the
  initial bundle and using `Suspense` for subapp namespaces.
- **Bengali typography fidelity.** Must load a real Bengali webfont; don't
  rely on system fallbacks.
- **Drift.** Without `i18next-parser` in CI, `bn.json` will slowly lag
  `en.json`. Make the extractor script mandatory before merging.
- **Capacitor origins.** `/locales/**` must be served from the same origin
  as `CAP_SERVER_URL` (or explicitly allowed via CORS). Not an issue today
  because both point at `https://toolkit.naurinjahan.com`.

## 11. Open questions

- Do we want a one-click "switch to Bengali" promo on first visit if
  `navigator.language` is `bn*`, or always default to English and let the
  user toggle? Current plan: let the detector pick `bn` if the browser
  reports it, with English fallback.
- Do we want per-user locale **server-side** now (Phase 4) or only when
  localized emails become a need? Current plan: wait.
